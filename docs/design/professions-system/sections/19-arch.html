Architecture (how it lands in this repo)

The system follows the repo's existing seams. Nothing below invents a new one.

Determinism and authority (hard invariants)

  • All gathering and crafting logic lives in src/sim/ and obeys the sim invariants: no DOM/Three imports, all randomness through Rng, fixed 20 Hz tick, no Math.random / Date.now / performance.now. Per-player node respawn timers, corpse-harvest state, effect durability/recharge, mobile crafting stations, and re-spec timers advance on the sim clock.
  • The server is authoritative. Harvest rolls, quality rolls, recipe resolution, material consumption, the self-gathered bonus, signed-material stamping, effect-durability consumption rolls, effect recharge, focus re-spec, and skill changes resolve server-side. The client streams intent and renders results.

Content-as-code (src/sim/content/, merged by data.ts)

  • professions.ts: the ten-craft wheel (the ring order and the per-slot tier ceilings), the archetype names, craft outputs (permanent / consumable / weapon lines), gathering profession definitions, focus and component-type configuration, specialization perks, base tool tiers, and the effect catalog.
  • recipes.ts: recipe records (inputs, craft + skill requirement, output, base ilvl budget, crafting-only effects, per-recipe binding policy).
  • gather_nodes.ts: world-node definitions and placements; monster component-type tags live on mob records.
  • Attunement quests (the per-archetype lore quest and the repeatable make-amends switch quest) are quest content; new craft and gather quest objective types let quests credit crafting and gathering.
  • Fungible materials and base tools are item records, so they flow through the existing inventory, vendor, and market systems unchanged.

Per-item-instance model (a foundation, built early)

Inventory today is itemId + count and items carry no identity. Signed materials, charged tools, rolled-quality crafted gear, and bind-to-recipient gear all need per-item identity: an instance id plus a payload (gatherer signature, per-effect charges, rolled quality/stats, the boundTo recipient). This is cross-cutting (inventory, equipment, trade, market, vendor, the interest-scoped snapshot, the wire encoder, and JSONB load/save) and is spec'd as its own early slice before tools, monster harvesting, and crafted gear. Fungible stacks (raw materials, infinite-durability base tools) keep flowing as itemId + count and do not need it.

New sim system behind the SimContext seam (src/sim/professions/)

  • gathering.ts: node harvest, per-player respawn state, corpse harvest, proficiency gain and rarity rolls.
  • crafting.ts: recipe resolution, material consumption, quality roll, ilvl budget computation, crafting-only effect application.
  • wheel.ts: the one-active-specialty mastery model (additive skill gain, the at-most-two active cap, the active archetype and attunement, the adjacent-pair majors and capped opposite hobby, specialization-concentration readout). Attunement switching is quest-driven (see Production core), not decay.
  • focus.ts: the additive gathering focus allocation plus the time/coin/material re-spec.
  • tools.ts: base-tool tier gating plus effect application, depletion, and recharge.

Per the seam rules, these modules hold functions; backing state (per-player craft skills, the active archetype and attunement history, the current hobby, focus allocations, respawn timers, corpse flags, re-spec timers) lives on Sim as live ctx views, with thin same-named delegates where a foreign caller resolves a method.

The IWorld seam, client, persistence

  • IWorld (src/world_api.ts): extend first, implement in both Sim and ClientWorld, then consume from render/ui. New surface: read craft skills and the wheel, current majors/hobby, specialization perks, focus allocation, node availability, corpse component options, recipe lists and craft results, tool tier and effect charges.
  • Client (render / ui): a professions/wheel HUD window following the module-first UI rules. All player-visible strings are t() keys; numbers/money through the formatters. Sim/server emit stable keys plus values re-localized by the matcher.
  • Guide / wiki: player-facing professions, gathering, and recipe content feeds /wiki; add guide.* prose keys and run npm run wiki:content.
  • Persistence (server/): per-player craft skills, the active archetype, attunement history (which archetypes have been entered and the make-amends count per archetype), current hobby, focus allocations, and learned recipes persist (JSONB on the character) with additive back-compat load. Signed-material gatherer names, per-effect charges, rolled quality/stats, and the boundTo recipient persist on item instances. Respawn timers and mobile stations are sim runtime state.
